# DestinyScope V1.3 Model Distribution Plan

更新日期：2026-05-29

## 1. 内置模型方案

内置模型指将 GGUF 或其他本地模型文件随 App Bundle 分发。

优点：

- 无下载流程。
- 断网可用。
- 用户不需要手动导入。
- 模型版本与 App 版本一致，便于复现问题。

风险和成本：

- 包体明显增大。当前 PoC 模型约 `491.4 MB`，会显著影响下载和安装。
- App Store 审核可能关注包体、AI 表述、模型用途和安全边界。
- 模型更新困难，需要重新发版。
- 必须确认模型 license、notice、商业使用、再分发和 App 内分发条件。
- 如果模型文件来自第三方量化仓库，需要确认量化文件本身的来源和许可说明。

适用条件：

- 完成 license / notice 审计。
- 完成多设备性能和内存测试。
- 明确最低设备要求。
- App Store 隐私政策和审核备注已更新。
- App 内或随包文档提供开源许可 / notice。
- 未完成 license / notice 审计前，不允许内置或分发模型。

## 2. 本地手动导入方案

本地手动导入指开发者或内测用户将 GGUF 文件手工导入 App 沙盒。

优点：

- 不增加 App 包体。
- 不需要服务端或 CDN。
- 适合 Debug / TestFlight 内测继续验证。
- 可快速切换模型文件做性能和质量对比。

缺点：

- 不适合正式用户。
- 操作门槛高。
- 模型来源和版本不易统一。
- 不适合作为 App Store 正式生产体验。

阶段结论：

- V1.3 可继续使用手动导入作为内测和验证手段。
- 不建议面向正式用户开放手动导入。
- TestFlight 也必须遵守 license / notice，不得使用不明来源模型。
- 即使模型不进入 App Bundle，也需要记录模型来源、license 和人工确认结果。

## 3. 按需下载方案

按需下载指 App 在用户开启本地模型功能时下载模型文件。

必要能力：

- 服务端或 CDN。
- sha256 校验。
- 断点续传。
- 下载失败恢复。
- 存储空间检查。
- 模型版本管理。
- 文件完整性校验。
- 隐私政策和网络说明更新。
- 异常处理和用户可理解的错误文案。
- license / notice 展示和随包说明。

优点：

- App 初始包体小。
- 可按设备、地区、版本或用户开关控制模型下载。
- 模型可独立更新，不一定依赖 App 发版。

风险：

- 当前产品原则是无服务端，下载方案会引入新的基础设施和审核说明。
- 需要处理网络失败、下载中断、存储不足和模型校验失败。
- 用户隐私政策需要说明网络请求和下载文件。
- App Store 审核可能关注模型内容和 AI 功能说明。

阶段结论：

- V1.3 暂不做正式模型下载。
- 如果未来选择该路线，应另开服务端/CDN、隐私、下载安全和模型版本管理专项设计。
- 模型下载或内置都需要 license / notice，不得绕过开源声明。

## 4. 推荐阶段性结论

V1.3 阶段不做正式模型下载。

推荐策略：

- 继续使用手动导入或 TestFlight 内测模型准备。
- 生产前再决定内置还是下载。
- 如果只做 TestFlight 内测，可延续 Debug / 内测导入流程。
- 如果准备 App Store 生产，需要先完成 license、notice、隐私、审核、设备分级和安全回退设计。
- 未完成 `docs/V1_3_LicenseNoticeAudit.md` 中的人工确认项前，不允许把模型或 `llama.xcframework` 随 App 分发。

当前不应提交模型文件到 git，也不应把 `llama.xcframework` 或 GGUF 放入 App Bundle。
