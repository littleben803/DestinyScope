# DestinyScope V1.2 本地小模型预研

更新日期：2026-05-27

## 1. V1.2 定位

DestinyScope V1.2 的目标是本地 0.5B 到 1.5B 小模型 PoC，不直接进入生产路径。

本地小模型在 V1.2 中只用于：

- 润色模板结果。
- 改善命理问答表达。
- 验证离线推理在 iPhone 真机上的可行性。
- 评估包体、内存、耗电、速度和文案安全边界。

本地小模型在 V1.2 中不得负责：

- 命理计算。
- 农历转换。
- 称骨权重查询。
- 称骨诗文匹配。
- 生成最终命理结论。
- 编造四柱、五行、十神、大运、流年等当前规则引擎未生成的数据。
- 突破免责声明和安全边界。

核心原则：规则引擎生成结论，模板系统生成基础文案，小模型只做表达润色。任何模型输出都必须受安全规则约束，并可回退到模板结果。

## 2. 技术路线对比

### 2.1 llama.cpp + GGUF

| 维度 | 评估 |
| --- | --- |
| iOS 接入难度 | 中等偏高，需要编译 C/C++ 推理库、处理 Swift/ObjC/C++ 桥接、线程和内存管理。 |
| 模型生态 | GGUF 生态成熟，社区模型和量化版本较多。 |
| 0.5B～1.5B 模型适配 | 适配空间较大，小模型和 4-bit 量化模型选择多。 |
| 量化支持 | 强，GGUF 常见 Q4/Q5/Q8 等量化格式。 |
| 包体大小 | 取决于模型；0.5B 4-bit 仍可能约数百 MB，1.5B 可能更大。 |
| 内存压力 | 中到高，需要真机测峰值内存和上下文长度。 |
| 流式输出 | 支持，但需要自行封装回调和 UI 调度。 |
| 设备兼容 | 覆盖面较好，但老设备性能和内存压力明显。 |
| App Store 审核风险 | 主要来自包体、AI 宣传、隐私说明和输出安全；技术本身可行但需谨慎描述。 |
| 后续生产化难度 | 中等，需要稳定模型选择、性能策略、安全过滤和降级机制。 |

结论：最适合 V1.2 PoC。优点是生态和量化成熟，缺点是工程集成、包体和性能压力需要实测。

### 2.2 Core ML

| 维度 | 评估 |
| --- | --- |
| iOS 接入难度 | 中等。系统框架接入友好，但模型转换、tokenizer、动态生成和采样链路不一定简单。 |
| 模型生态 | App 侧生态较规范，但 LLM 可用模型和转换链路需要单独验证。 |
| 0.5B～1.5B 模型适配 | 理论可行，实际取决于模型转换质量、算子支持和设备性能。 |
| 量化支持 | 支持 Apple 生态内的量化和压缩路线，但 LLM 部署细节需要专项验证。 |
| 包体大小 | 仍取决于模型，可能较大。 |
| 内存压力 | 需要真机验证，可能受系统调度影响。 |
| 流式输出 | 需要自行设计生成循环和 token 输出。 |
| 设备兼容 | Apple 平台一致性较好，但系统版本和硬件能力会限制体验。 |
| App Store 审核风险 | 系统框架路径相对稳妥，但仍需要隐私、元数据和输出安全说明。 |
| 后续生产化难度 | 中到高，重点在模型转换、tokenizer、推理速度和调试效率。 |

结论：适合长期评估，尤其是希望走 Apple 原生推理和系统能力路线时。但 V1.2 第一轮 PoC 不建议从 Core ML 起步。

### 2.3 MLX Swift

| 维度 | 评估 |
| --- | --- |
| iOS 接入难度 | 中等到高，需要确认 MLX Swift 在目标 iOS 环境、包管理和发布链路中的成熟度。 |
| 模型生态 | Apple Silicon 生态增长快，但 iOS 生产案例需要进一步调研。 |
| 0.5B～1.5B 模型适配 | 有潜力，尤其适合 Apple 设备侧实验。 |
| 量化支持 | 需要按具体模型和 MLX 当前能力确认。 |
| 包体大小 | 取决于模型和运行时依赖。 |
| 内存压力 | 需要真机验证，尤其是 iPhone 而不是 Mac。 |
| 流式输出 | 可设计，但需要看 SDK 能力和示例成熟度。 |
| 设备兼容 | 可能更依赖较新的 Apple 芯片和系统版本。 |
| App Store 审核风险 | 需要评估依赖稳定性、包体、AI 宣传和隐私说明。 |
| 后续生产化难度 | 中到高，取决于 iOS 支持成熟度和调试工具。 |

结论：适合作为备选路线，尤其用于验证 Apple 生态本地模型能力。但在 V1.2 第一轮 PoC 中优先级低于 llama.cpp + GGUF。

## 3. 推荐 PoC 路线

首选路线：llama.cpp + GGUF。

理由：

- GGUF 小模型和量化生态较成熟。
- 0.5B 到 1.5B 的模型选择更多。
- 更容易做离线 PoC 和横向模型对比。
- 可以先做 Debug-only 集成，不影响生产路径。

备选路线：MLX Swift。

理由：

- Apple 设备侧潜力大。
- 若后续 MLX Swift iOS 支持成熟，可减少部分 C++ 集成复杂度。

长期评估：Core ML。

理由：

- Apple 官方生态更适合长期生产化。
- 但第一轮 LLM PoC 的转换、tokenizer、采样和调试成本较高。

## 4. 候选模型筛选标准

候选模型必须满足：

- 参数规模在 0.5B 到 1.5B。
- 支持中文输入和中文输出。
- 支持基础指令跟随。
- 可量化到 4-bit。
- 可完全离线运行。
- 输出稳定，重复测试不频繁越界。
- license 允许 App 使用，需要人工确认商业和分发限制。
- 包体和内存可接受。
- 可限制输出长度，适合 200 到 500 字短回答。
- 对安全规则有较好遵循能力。

排除标准：

- license 不清晰或不允许 App 分发。
- 中文能力明显不足。
- 需要联网服务。
- 仅支持服务端推理。
- 量化后质量不可接受。
- 低端 iPhone 上加载或生成容易崩溃。

## 6. 风险

技术风险：

- 包体过大，影响下载和上架体验。
- 首次加载慢，影响用户耐心。
- 内存峰值高，低端设备可能崩溃。
- 耗电和发热明显。
- 老设备不可用或体验不可接受。
- 模型输出不稳定。
- 模型可能出现幻觉，编造命理结论。

产品和合规风险：

- 输出可能越过免责声明和安全边界。
- 输出可能生成医疗、法律、投资、婚恋等现实决策建议。
- App Store 元数据不能夸大 AI 能力。
- 如果进入生产，需要更新隐私政策，说明本地模型处理方式。
- 如果模型文件随 App 分发，需要确认模型 license、版权和再分发限制。
- 如果未来支持模型下载，需要重新评估隐私、网络、存储和审核风险；V1.2 PoC 不做模型下载。
