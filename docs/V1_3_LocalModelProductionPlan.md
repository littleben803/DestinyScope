# DestinyScope V1.3 Local Model Production Plan

更新日期：2026-05-29

## 1. V1.3 定位

V1.3 从 V1.2 的 Debug-only PoC 进入生产化方案设计阶段。

本阶段不是直接把本地模型接入正式结果页，也不是替换当前模板输出。V1.3 的目标是先确定：

- 模型分发策略。
- 设备分级策略。
- 隐私政策和 App Store 审核说明。
- 输出安全边界。
- 模型失败时的模板回退策略。
- 是否允许进入 TestFlight 内测。

当前 Release 仍应保持本地规则引擎和模板输出。本地模型只能作为未来的文本润色层，不负责命理计算、农历转换、称骨权重、诗文匹配或最终命理结论。

## 2. 生产化候选模式

### 模式 A：继续纯模板，不上本地模型

优点：

- 最稳。
- 包体小。
- 审核风险低。
- 隐私政策和审核备注变化最少。
- 老设备体验一致。

缺点：

- 互动性有限。
- 文案自然度有限。
- 难以体现本地小模型 PoC 的价值。

### 模式 B：本地模型作为可选实验功能

特点：

- 默认关闭。
- 用户手动开启。
- 只在高性能设备显示。
- 模型失败自动回退模板。
- 模型只用于 `TextRefining` 润色。
- 不生成命理结论。

这是 V1.3 的首选方向。

优点：

- 风险可控。
- 不影响当前主流程。
- 可以通过 TestFlight 收集真实设备反馈。
- 设备不满足条件时继续使用模板。

缺点：

- 需要设计开关、设备分级、文案说明和回退策略。
- 仍需补齐 license、隐私和审核材料。

### 模式 C：本地模型默认用于结果页润色

特点：

- 默认开启。
- 直接影响用户主流程。
- 结果页文案默认经过本地模型润色。

风险：

- 老设备体验可能变慢。
- 生成失败或安全回退会影响主流程体验。
- App Store 元数据、隐私政策和审核备注必须同步更新。
- 模型输出质量和安全性需要更高保障。

结论：暂不建议。

### 模式 D：模型文件内置 App Bundle

优点：

- 无下载流程。
- 断网可用。
- 用户不需要手动导入。
- 模型版本和 App 版本绑定，调试相对简单。

缺点：

- 包体明显增大。
- App Store 审核和分发成本更高。
- 模型更新困难，需要发版。
- license、notice、再分发和 App 内分发必须完全确认。

### 模式 E：模型文件按需下载

优点：

- App 初始包体小。
- 可按设备或用户开关下载。
- 模型可独立更新。

缺点：

- 需要服务器或 CDN。
- 需要 sha256 校验。
- 需要断点续传。
- 需要存储空间检查。
- 需要失败恢复和版本管理。
- 隐私政策和网络说明变复杂。

结论：暂不适合当前阶段。

## 3. 推荐方案

V1.3 推荐采用模式 B 的生产化设计方向：

- 暂不进入默认主流程。
- 先设计“本地润色实验开关”。
- 仅 Debug / TestFlight 内测开放。
- 默认关闭。
- 用户手动开启。
- 首次开启必须展示实验说明和用户确认。
- 只用于 `TextRefining` 润色。
- 不生成命理结论。
- 不替代 `LifeWeightEngine`、`TemplateFortuneInterpreter` 或 `TemplateFortuneQuestionAnswerer`。
- 模型失败、安全检查失败或设备不满足条件时，回退 `TemplateTextRefiner`。

该方案保留 V1.2 PoC 的技术价值，同时避免直接影响 Release 用户路径。

阶段 3 结论：

- TestFlight 实验开关是 V1.3 推荐路线。
- 开关默认关闭，用户手动开启。
- 开关只影响 `TextRefining` 润色和可选的润色预览。
- 开关不影响称骨计算、农历转换、诗文匹配、`LifeWeightResult`、`LifeWeightInsight`、命理问答结论、历史记录原始结果或知识库内容。
- App Store Release 默认不展示该实验开关，除非后续单独决策。
- 详细策略见 `docs/V1_3_TestFlightExperimentPlan.md`。

## 4. 设备分级策略

已知 benchmark：

- iPhone 17 Pro Max：约 1 秒级，体验可接受。
- iPhone 12 mini：部分正常样例约 4 到 5 秒，可运行但不适合默认开启。

因此 V1.3 设备分级策略必须重点考虑 A14 / 小内存设备默认关闭或仅实验开启。

阶段 2 结论：

- V1.3 不应全量默认启用本地模型。
- 推荐先只对 Tier A 高性能设备开放 TestFlight 实验开关。
- iPhone 17 Pro Max 可作为 Tier A 参考设备，适合进入内测验证。
- iPhone 12 mini benchmark 支持“不能默认全量开启”的判断。
- iPhone 12 mini / A14 级别设备应归入“谨慎启用 / 默认关闭 / 低端禁用”候选。
- 任何无法识别的设备默认禁用本地模型，继续使用模板。
- 详细策略见 `docs/V1_3_DeviceTierPlan.md`。

后续需要：

- 记录具体 iPhone 型号。
- 至少按内存和芯片分级。
- 高性能设备允许显示实验开关。
- 低端设备隐藏或禁用本地模型。
- 老设备只使用模板。
- 继续补充 iPad、低端 iPhone、连续运行和发热 benchmark。
- 使用 Xcode Instruments 记录 Memory、CPU、Energy 和 Thermal。

初步建议：

| 设备等级 | 策略 |
| --- | --- |
| 高性能设备 | 可在 TestFlight 中显示实验开关，默认关闭 |
| 中端设备 | 可评估是否显示实验开关，默认关闭 |
| A14 / 小内存设备 | 谨慎或默认关闭，只使用模板 |
| 更老设备 | 隐藏本地模型能力，只使用模板 |

初步超时建议：

- Tier A：生成超过 3 秒回退模板。
- Tier B：生成超过 5 秒回退模板。
- Tier C：不启用本地模型。

## 5. 回退策略

任何情况下，本地模型都不能成为唯一输出路径。

必须回退模板的场景：

- 模型不存在：模板。
- 加载失败：模板。
- 生成超时：模板。
- 安全检查失败：模板。
- 输出为空：模板。
- 输出过长：模板。
- 输出包含高风险词：模板。
- 输出包含提示词泄露或特殊 token：模板。
- 用户关闭开关：模板。
- 设备不满足条件：模板。

回退时应保持用户体验稳定，不展示内部错误栈，不让用户看到模型路径、框架错误或调试日志。

## 6. 安全边界

本地模型只能润色已有文本，不能扩大事实范围。

明确禁止：

- 新增命理结论。
- 新增四柱、五行、十神、大运、流年等当前结构化结果中没有的数据。
- 进行医疗、法律、投资、婚恋、职业确定性建议。
- 输出精准预测、改命、化解、避灾、必然发财、保证转运、寿命疾病预测、疾病预测、投资收益确定、婚姻确定性等内容。
- 以“大师”“真实 AI 算命”等形式包装能力。

所有生产化设计都必须保持：命理计算和结论由本地规则引擎生成，本地模型只负责表达润色。

## 7. 阶段结论

V1.3 不应直接默认启用本地模型。推荐先完成 TestFlight 内测开关、设备分级、隐私审核材料、license notice 和安全回退方案，再决定是否进入生产代码改造。
