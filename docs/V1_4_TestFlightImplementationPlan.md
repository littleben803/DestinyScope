# DestinyScope V1.4 TestFlight Implementation Plan

更新日期：2026-05-29

## 1. V1.4 定位

V1.4 从文档设计进入 TestFlight 本地模型内测实现准备。

边界：

- 不进入 App Store 正式生产。
- 不默认开启本地模型。
- 不替换默认结果页。
- 不把模型输出作为命理结论。
- 只实现受控 TestFlight 实验路径。
- 继续保持 `TemplateTextRefiner` 作为默认输出路径。

V1.4 阶段 1 仍只做实现拆解，不直接修改 Swift 代码。

## 2. V1.4 目标能力

需要逐步实现：

- TestFlight / Debug 实验开关。
- 设备 tier 检测。
- 本地模型可用性检测。
- 低电量 / 过热 / 超时回退。
- 结果页底部“本地润色预览”卡片。
- 用户手动触发润色。
- 安全检查失败回退模板。
- App 内开源许可页面草案。
- 隐私页面更新草案。
- TestFlight 测试说明。

所有能力都必须是受控实验路径，不得改变当前默认结果页输出。

## 3. 非目标

V1.4 明确不做：

- 不做 App Store 正式开关。
- 不做默认启用。
- 不做模型下载。
- 不做云端 AI。
- 不做 RAG。
- 不做流式 UI。
- 不做开放聊天。
- 不做付费订阅。
- 不把模型输出作为命理结论。
- 不保存润色结果到历史记录。
- 不上传用户数据。
- 不上传模型输入输出。

## 4. 实现拆解

### 阶段 A：实验开关和配置模型

- 新增 `LocalModelExperimentSettings`。
- 默认关闭。
- 保存用户是否开启。
- 仅 Debug / TestFlight 可用。
- Release 正式版默认隐藏或不可用。
- 开关不能改变默认 `TextRefinerFactory.makeDefaultRefiner()`。
- 开关只能影响受控实验入口。
- 首次开启需要展示实验说明。

### 阶段 B：设备 tier 检测

- 新增 `DeviceTierService`。
- 识别设备 model identifier。
- 判断 Tier A / Tier B / Tier C。
- 未识别设备默认禁用或归入谨慎。
- 低电量模式 / thermal state 作为运行时禁用条件。

初始策略：

- Tier A：允许 TestFlight 实验开关。
- Tier B：谨慎显示或提示可能较慢。
- Tier C：默认不显示或不可用。

### 阶段 C：模型可用性检测

- 检查模型文件是否存在。
- 检查 llama framework 是否可用。
- 检查模型大小。
- 不自动下载。
- 不复制到 Bundle。
- TestFlight 阶段仍优先使用手动导入或开发者准备方式。
- 不将模型提交仓库。
- 不将模型打入正式 Release。

### 阶段 D：结果页润色预览卡片

- 只在实验开关开启、设备支持、模型存在时展示。
- 用户手动点击“生成润色版”。
- 原文始终保留。
- 润色结果仅当前页面展示。
- 不写入历史记录。
- 展示位置为结果页底部。
- 标题建议：“本地润色预览”。
- 说明本地模型只做文本润色，不生成新的命理结论。

### 阶段 E：超时和回退

- Tier A：3 秒超时。
- Tier B：5 秒超时。
- Tier C：不启用。
- 失败回退 `TemplateTextRefiner`。
- 安全检查失败回退原文。

回退场景：

- 模型不存在。
- framework 不可用。
- 加载失败。
- 生成超时。
- 输出为空。
- 输出过长。
- 安全检查失败。
- 低电量模式。
- 过热状态。
- 连续失败次数过多。

### 阶段 F：开源许可和隐私说明

- About 页面增加“开源许可”入口草案。
- 隐私政策页面更新草案。
- TestFlight 说明文案。
- 正式上线前仍需人工确认 license / notice。
- 如果 TestFlight 包含模型或 framework，需要开源许可说明可见。

## 5. 数据与隐私边界

明确：

- 不上传出生信息。
- 不上传命理结果。
- 不上传模型输入输出。
- 不上传历史记录。
- 不记录完整润色文本。
- 不做远程日志。
- 只允许本地记录开关状态和必要运行状态。

允许本地记录的最小运行状态：

- 实验开关是否开启。
- 设备 tier。
- 最近一次是否 fallback。
- 错误类型。
- 简短耗时指标。

不得记录：

- 完整模型输入。
- 完整模型输出。
- 用户出生信息。
- 完整命理结果。
- 可识别个人身份的信息。

## 6. 模型文件策略

V1.4 初稿建议：

- TestFlight 阶段仍不内置模型。
- 继续使用手动导入或开发者准备方式。
- 不做下载。
- 不把模型文件提交仓库。
- 不把模型打入 Release。
- 进入真正 TestFlight 前必须再次确认 license / notice。

未完成 license / notice 人工确认前，不允许随 App 分发模型。

## 7. 构建策略

- Debug 保留完整调试入口。
- TestFlight 可显示简化实验开关。
- App Store Release 默认隐藏。
- 任何情况下 `makeDefaultRefiner()` 仍返回 `TemplateTextRefiner`，除非用户在受控实验路径手动触发。
- Release 构建不得暴露本地模型入口。
- 模型文件和 `llama.xcframework` 不得误提交仓库。

## 8. 验收标准

V1.4 最终验收：

- Debug 构建通过。
- Release 构建通过。
- 正式 Release 不暴露实验入口。
- 默认输出路径不变。
- Tier C 不显示本地润色入口。
- 模型不存在时回退。
- 低电量 / 过热时回退。
- 超时时回退。
- 安全检查失败时回退。
- 原始结果不被覆盖。
- 历史记录不保存润色结果。
- 仓库不包含模型文件或 xcframework。

## 9. 阶段结论

V1.4 可以开始做 TestFlight 本地模型内测的实现准备，但第一步应继续保持拆解和技术设计优先。任何实现都必须小步推进，并保持默认输出路径不变。
