# DestinyScope V1.5 UI / UX 全量审计报告

检查日期：2026-06-01

## 1. 总体结论

当前 App 已具备完整产品结构：根 Tab、首页输入、结果页、命理问答、知识库、历史记录、设置 / 关于、隐私政策、免责声明、开源许可，以及受控的本地模型实验路径均已形成闭环。

当前最大 UX 风险：

- 结果页内容密度高，称骨结果、洞察、问答、解读、本地润色卡片和免责声明都集中在一个长滚动页面中，后续需要更清晰的分区层级、折叠 / 展开或导航锚点。
- 知识库已扩充到 29 篇，但列表缺少分类筛选和搜索，用户需要完整滚动才能找到主题。
- 历史记录只提供列表，不提供详情页；用户很难回看完整诗文和结果上下文。
- 本地模型实验入口虽然受控，但设置页和 Debug 页信息量大，仍需要避免用户误以为模型参与命理计算或生成结论。
- Legal / 隐私 / 开源许可内容较长，当前为连续卡片，需要更清楚的目录或分区摘要。
- 小屏、深色模式、Dynamic Type 和 VoiceOver 需要人工验证，尤其是长路径、长按钮、长卡片和暗金 / 朱砂对比度。

最值得优先优化的页面：

1. 结果页。
2. 知识库列表 / 详情。
3. 历史记录。
4. 设置 / Legal / 本地模型实验入口。
5. 可访问性、深色模式、小屏适配。

## 2. 页面审计

### 根导航

- 页面名称：根导航
- 文件路径：`DestinyScope/UI/MainContentView.swift`
- 当前功能：使用 `TabView` 承载首页、知识库、历史、关于四个入口。
- 主要问题：
  - “关于” Tab 同时承载设置、Legal、本地模型实验和 App 说明，语义略窄。
  - 本地模型 Debug 入口只在 Debug 下展示，但普通信息入口和实验入口都集中在同一个 Tab。
- 优先级：P1
- 建议优化：
  - V1.5 阶段 7 评估 Tab 名称是否保持“关于”或改为“设置”。
  - 保持四 Tab 结构，不新增复杂导航。
- 是否需要代码改动：是
- 是否影响隐私 / 审核：低
- 是否影响本地模型默认路径：否

### 首页

- 页面名称：首页
- 文件路径：`DestinyScope/UI/Home/HomeView.swift`
- 当前功能：出生日期选择、时辰选择、查询按钮、错误提示、本地隐私提示。
- 主要问题：
  - DatePicker 和时辰 Picker 在小屏上可能显得拥挤。
  - 时辰仅显示 `00 时` 到 `23 时`，没有解释传统时辰概念，可能与用户预期的“子时、丑时”等不一致。
  - 错误提示为通用卡片，缺少下一步指引。
  - 隐私提示足够克制，但可考虑更接近输入区。
- 优先级：P1
- 建议优化：
  - V1.5 阶段 8 检查小屏和 Dynamic Type。
  - V1.5 后续可增加传统时辰辅助说明，但不要改变计算逻辑。
  - 错误态增加“请检查日期和时辰后重试”等行动文案。
- 是否需要代码改动：是
- 是否影响隐私 / 审核：低
- 是否影响本地模型默认路径：否

### 结果页

- 页面名称：结果页
- 文件路径：`DestinyScope/UI/Result/DestinyResultView.swift`
- 当前功能：展示命格标题、农历生日、总重量、诗文、权重明细、命格洞察、命理问答、命理师解读、本地润色预览卡片和免责声明。
- 主要问题：
  - 内容顺序和层级偏长：洞察、问答、解读都很重要，但用户首次进入容易被长页面压住。
  - 权重明细目前为简单 HStack，小屏下长文可能挤压。
  - `FlowLayout` 当前在 iOS 16+ 使用 `HStackLayout`，标签过多时可能仍横向拥挤，未真正自动换行。
  - 本地润色预览卡片在条件满足时插入结果页底部，说明文案克制，但仍需确保用户知道“原始结果不变”。
  - `safetyNotice` 分散在洞察、解读和底部，可能重复但不够醒目。
- 优先级：P0
- 建议优化：
  - V1.5 阶段 4 优先优化结果页阅读结构。
  - 增加区块摘要或折叠 / 展开：称骨结果、洞察、问答、解读、本地润色预览。
  - 强化“原始结果”和“润色预览”的视觉区隔。
  - 优化权重明细和标签换行。
  - 统一免责声明展示策略，避免重复又不削弱边界。
- 是否需要代码改动：是
- 是否影响隐私 / 审核：是，需保持免责声明可见
- 是否影响本地模型默认路径：否

### 本地润色预览卡片

- 页面名称：本地润色预览卡片
- 文件路径：`DestinyScope/UI/Components/LocalRefiningPreviewCard.swift`
- 当前功能：实验条件满足时展示，用户手动触发本地 TextRefining 润色；不覆盖原文、不写历史。
- 主要问题：
  - 卡片内容包含原始文本摘要、润色结果、engine、wasRefined、耗时等调试字段，对普通 TestFlight 用户可能偏技术化。
  - “使用原始文本”会收起预览，但用户可能不理解是否关闭了实验。
  - 原始摘要 `lineLimit(6)` 对长文本较克制，但仍可能占空间。
- 优先级：P0
- 建议优化：
  - V1.5 阶段 4 或 7 将技术字段仅 Debug 展示，TestFlight 简化为结果、状态和温和提示。
  - 明确“原始结果已保留，本卡片仅为临时预览”。
  - 保持用户手动触发，不自动生成。
- 是否需要代码改动：是
- 是否影响隐私 / 审核：是，需防误解
- 是否影响本地模型默认路径：否

## 3. P0 问题列表

1. 结果页内容过长，分区层级需要折叠 / 锚点 / 更清晰标题。
2. 知识库 29 篇文章缺少分类筛选或搜索，浏览效率不足。
3. 历史记录缺少详情页或更清楚的本地保存说明，清空全部缺少确认。
4. 本地模型实验入口需要继续避免“AI 算命”误解，TestFlight 体验应隐藏过多技术字段。

7. 深色模式和暗金 / 朱砂对比度需要人工检查。
8. VoiceOver 标签和 Dynamic Type 支持需要系统性检查。
